Activity-led reward

How big is the base** mint at confirm — EMA, LP cap, manipulation discount — before any escrow supplement or green-light?

Story context

  • Book ch 10 · Issuance & positive blocks

Positive-block size follows network activity, not a fixed schedule or remainingSupply() curve.

Design: Adaptive issuance · anchor: The positive block.

Not here: depression floor, green-light fresh mint, or other pay on top of this reward — Issuance network states.

Live formula (84532)

On each confirm, IssuanceModule computes:

activityMint = emaSlow × emaToMintBps / 10_000          (default emaToMintBps = 2500)
reward       = min(activityMint, mintLpCapBps × lpBalance / 10_000)   when LP > 0
             = activityMint                                            when LP = 0

Default mintLpCapBps = 500.

This reward is what splits to auditor / treasury / founder at confirm (Where mint splits). Supplements in depression are separate draws — ch 05.

Dual EMA

EMAMeaning
`emaFast`Recent bounty-weighted activity — reacts quickly
`emaSlow`Smooth baseline — drives mint size

Bounty signal is protocol-credibility-weighted (kProtocol blend vs network mean). Auditors below emaSlowMinSuccessfulForFullWeight successful audits contribute at reduced weight until full weight earned.

Manipulation discount

When emaFast / emaSlow > manipulationThresholdBps → Manipulation → base mint scaled by manipulationMintScaleBps (default 8000 = 80%).

Damps suspicious spikes on the positive-block reward — settlement still runs; only size changes.

Worked examples (defaults)

A — activity binds: emaSlow = 10_000 → activityMint = 2_500; LP cap 5_000 → reward = 2_500

B — LP cap binds: activityMint = 12_500; cap 5_000 → reward = 5_000

C — manipulation: 2_500 × 0.8 → 2_000

Integrators: preview = nextPositiveBlockReward(); history = auditBlockRewardMinted(id).
