Verify it yourself

A verdict is a proof, not an opinion. Recompute any settled result from the live code — no wallet, no account, no tokens.

Anyone can check

You don't need to be registered or hold anything to verify an audit. If a result is honest, you can re-derive it yourself from the live, on-chain code — in under a minute, on any computer.

One click, in this tab

Checking used to cost something: install node, open a terminal, copy a command. Now every audit in the explorer has a Reproduce it now button — click it, and your own browser does the recomputation and confirms the match. Nothing to install, no wallet, no account; the page only reads from the chain.

You'll see the same story the command tells. First a few MATCH ✓ lines — the checker proving its own math before it checks anything else. Then one of two endings: VERIFIED ✓ means the recorded verdict really is the honest pass/fail of the code deployed on chain right now. MISMATCH ✗ means it isn't — wrong contract, code that changed since the audit, or an audit that used a different tool than the standard one this checker knows. The button never rounds a mismatch up to a pass. A successful run is remembered on this device as reproduced by you.

If you watch audits (the ☆ in the explorer), the Your audits rail does two more things for you. It reads each watched audit's state live from the chain on every visit and flags anything that moved since you last looked — "AwaitingWindow → Claimed since your last look". And one Re-verify all watched button runs this same reproduction over your whole watchlist and updates your personal ledger, so a morning glance covers everything you care about. Same honest limits: read-only, this device only, and audits newer than the site's snapshot are skipped until the index refreshes. The summary also says why a non-match isn't an alarm when the chain already explains it: "not settled yet" (no verdict on chain to check), "different spec" (this in-tab checker only knows the canonical fixture recipe — use the CLI/harness), and "expected mismatch" (the verdict was overturned — the checker refusing to confirm it is the record working). The red MISMATCH is reserved for the one case that deserves your attention.

One honest note: the button lives on this site, and the whole point is that you shouldn't have to trust any site. If you'd rather trust nothing but your own machine, the command below proves the same thing from a terminal — button and command run the exact same code, one shared core.

One command

With node installed, from the membrane folder:

node verify.mjs 7

The script self-tests its own math against published values, fetches the audited contract's live bytecode, recomputes the result, and compares it to what the cell recorded.

What you'll see

A row of MATCH ✓ lines — the script proving its own encoding — then VERIFIED ✓ with the settled verdict. Try audit #6 and you'll get a MISMATCH: correct, because that result was overturned. Verify refusing to confirm a bad result is exactly the point.

The honest limit

This re-derives the encoding and its binding to the real deployed bytecode — that the recorded result is the honest encoding of pass/fail on the actual code, under the named spec and tool. It does not re-execute the contract in an EVM; independently re-running the fixture to re-derive the verdict from scratch needs a local EVM fork (the harness). The tool says which layer it proves — it never overstates.

Why it matters

This is the whole point of the network. You never have to trust the auditor, the protocol, or this site — you recompute, and the numbers either match or they don't.
