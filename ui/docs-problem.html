Survival condition

Deployed contract bytecode is immutable. Incorrectness before deployment is permanent loss — often borne by users who never saw the audit report.

Correctness at launch is therefore a survival condition, not a quality target or marketing polish.

A design that fails any row below is not a valid implementation of this notebook’s mechanism target.

Required properties

Any valid mechanism must meet all of the following:

PropertyMeaning
BinaryPass or fail only — no severity spectrum, partial pay, or committee grade
Machine-checkableSame registered tool + same pinned code + same spec → same answer for any party
PermissionlessEligible participants join and act without a gatekeeper approving membership
Skin in the gameAuditor economics tie to long-run accuracy, not time-to-report or relationship preservation
Self-fundingRewards issue from mechanism output (escrow, mint rules), not client goodwill alone
AccumulatingEach settled audit adds to a shared, reusable toolkit of checks — network learns in public
Judgment-freeWho audits, what counts as a claim, when pay moves — deterministic from on-chain inputs + tool replay, not discretion

What this rules out

Out of scope for a valid mechanism
Qualitative severity threads that gate payout
“Trust the firm” as the final verification step
Payment that clears before accuracy is falsifiable on-chain
Admin or committee veto after tool proof is posted

Disputes that do belong in the live design collapse to reproducible pass/fail under registered tools (Flows book).

Implementation

The Audit Cell implements these constraints on-chain — L0 settlement core, L1 satellites for overlays, L2 organs for read models only.
