Tokenomics

Supply is a function of verified work. Here is the function.

AUDIT is the network's unit, and it has one law: it is created only by audits that survive scrutiny. There is no schedule, no cap, and no discretionary issuance. Everything below is that law written out — the mint sized from real activity, the split it pays into, the reserves that fund bug payouts without printing, and the demand that makes the unit worth holding.

How a block is minted

Every confirmed audit — one that passed, then went its whole challenge window with no one breaking it — mints exactly one block. The reward is not drawn from "what's left until a cap." It is sized from how much verified work the network is actually clearing, and then paced so it cannot outrun the liquidity behind it:

reward = min( emaSlow · 0.25 ,  lpBalance · 0.05 )     — while liquidity is on the book
reward =      emaSlow · 0.25                           — before any liquidity exists
reward → reward · 0.80                                 — if the activity burst looks manufactured

The first number is a quarter of the activity signal — how much verified work the network is clearing. The second is a liquidity brake: the mint can't exceed a small slice of the pool behind the token, so supply grows only as fast as the market backing it. The block pays whichever is smaller. And if activity spikes far above the slow trend — the mark of a faked burst — the reward is trimmed, so manufactured volume earns less, not more. Nothing here points at a maximum supply, because there isn't one.

The activity signal

The signal the mint reads from, emaSlow, is a slow running average of the bounties flowing through the network — but weighted by each poster's credibility, not the raw amount posted. A bounty moves the signal only as much as the poster's standing has earned:

emaSlow    ← EMA( credBounty )
credBounty = ( nEff · protoMean  +  10 · netMean ) / ( nEff + 10 )

A brand-new address barely moves the signal — its bounties get pulled back toward the network average. Only a track record built with many different, established partners (the nEff in the formula) lets a protocol's own volume start to count. That is what blocks the obvious cheat — a single operator posting bounties among their own wallets to pump the mint. Self-dealing never builds a history of distinct partners, so the weighting keeps that volume discounted.

Where each block goes

A single confirmed block mints in three directions at once. The auditor's reward is the base; the other two are sized from it:

auditor    reward
treasury   1.00 · reward   →  75.1% liquidity  +  24.9% escrow reserves
founder    0.0305 · reward  (≤ 15,000,000 AUDIT ever; released × min(1, audits / 1000))

The treasury share matches the auditor's one-for-one and splits inside the escrow: most of it deepens the liquidity pool (capped at 15% of supply), the rest locks into reserves under a 180-day timelock before aging into liquidity. The founder slice is deliberately small, hard-capped at fifteen million tokens for the life of the network, and released only in proportion to audits actually completed — it accrues as the network works, never as a genesis grant.

Bugs pay from reserves, never from mint

When a discoverer reproduces a bug on a contract that passed, they are paid — but not with new supply. The payout is drawn from the escrow reserves that earlier blocks already set aside:

boost  = 1 + 4 · max(0, failed − found) / (successful + failed + found)   ∈ [1×, 5×]
payout = min( bounty · boost ,  max( 0.05 · escrow ,  0.50 · bounty ) )
stake  = 0.20 · bounty      (posted to file a claim; refunded on resolution)

Δsupply | bug  =  0

The reward scales with the original bounty and with the assigned auditor's record — a worse track record directs more hunter attention, up to five times the bounty — but it is bounded by a fraction of the reserves and a floor on the bounty. Because failures spend reserves instead of minting, the most dangerous attack on any issuing system — stage a break, collect freshly printed reward, repeat — cannot print here. It is closed by construction, not by vigilance.

The cost of a seat

Auditors are drawn from an open pool; register() is permissionless. What keeps the pool from being flooded by sock puppets is a hold that rises with each seat taken:

hold(N) = (N − 1) · 1 AUDIT

The first auditor to register needs nothing — which is the only honest answer at genesis, when no tokens exist yet. Each later seat needs one more token already minted by real work, so the cost of standing up an army of identities grows with the army. The rule bootstraps itself and prices Sybils in the same line.

Genesis and the mint lock

There is no premine. The first AUDIT in existence is earned by the genesis audit clearing its window — the same path every later token takes. Once the network is running, the minting key is locked one way and permanently: from then on, only the wired issuance module can create supply, and only along the function above.

Why there is no hard cap

A fixed ceiling is a guess about a network's eventual size, and both ways of guessing fail. Set it low and a working network throttles against a wall it should never have hit; set it high and the number means nothing. An activity-led mint needs no such guess: it issues in proportion to verification actually happening, expands when real work expands, and quiets when work quiets. Inflation is held down not by a ceiling but by four things at once — only confirmed work mints, the liquidity brake caps each block, manufactured spikes are damped, and every failure path spends reserves instead of printing. The supply is unbounded the way a tally is unbounded: there is no largest value it is allowed to reach — only however much verified work has actually happened.

Two more mint paths

Two rare paths also mint, both mechanism-gated and neither touching bug payouts: when a verification tool crosses its canonical-use threshold, its author receives a one-time canonization mint; when a structural upgrade is adopted, its proposer receives an adoption mint. That is the entire supply story.

Where demand comes from

Issuance is only half of it. The unit is needed on both sides of the rail: to post a bounty a protocol must hold the token the network pays in, and to audit a participant must hold the position-scaled stake above. The token's value is not a claim about what an audit is "worth" in the abstract — it is the standing demand of everyone who needs to use the rail at all. Earned on one side, required on the other, and gifted to no one.
