Spec arbiter

Was the Gate A filing (spec + PASS digest on file) wrong or non-auditable — and should the line be voided before any positive block mints?

Story context

  • Book ch 12 · Overlays

The spec arbiter is the Gate A overlay: a permissionless challenge to the declared spec PASS binding while the audit row is still pre-mint. Settlement is binary tool replay — not a severity debate. The cell records digests and lane locks; SpecArbiterModule (L1 satellite) drives hooks into AuditCell.

Missed-bug fights on pinned code belong on Vulnerability disclosure (Gate C). Gate A declare + re-run: The door (Gate A). Lane lock: Audit mutex (Spec lane).

Module: SpecArbiterModule.sol · Cell hooks: settlementSpecChallenge lock / unlock / void.

                                        │

                    ▼                   ▼                   ▼
             declaration holds    Invalidated          liveness expiry
             (challenge loses)    (challenge wins)       (see flow)

Successful invalidation → `Invalidated`: bounty returned (per rules), artifact pin cleared, no positive block minted.

Who runs the arbiter

The assigned spec arbiter is the next eligible pool registrant at challenge open — not:

ExcludedReason
ProtocolSubmitted the spec at Gate A
Original auditorAttested PASS at accept
ChallengerHas stake in the outcome

Anyone may call `reassignSpecArbiter` if the drawn party is ineligible.

Flow

challengeSpecInvalid(auditId, failErrorsRoot)
        ↓
assign spec arbiter (or SpecArbiterUnavailable)
        ↓
  ┌─ arbiter assigned ── declareSpecArbitrament(errorsRoot)
  │       MATCH stored PASS digest → challenger slashed; audit continues
  │       NO MATCH → invalidate; fee split; challenger stake refunded
  ├─ arbiter silent ── expireSilentSpecArbiter
  ├─ no arbiter ── protocol defendSpecChallenge(passRoot)  [fast path]
  └─ no arbiter, window elapsed ── finalizeSpecChallenge → invalidate

Challenger files FAIL evidence (failErrorsRoot) against the declared spec binding. Defense / arbiter replays PASS (or FAIL) and compares to `specPassDigest` stored at submit.

While the Spec lane is open: blocks acceptAudit, in-audit verdict, confirmAudit, claimVulnerability, and new dispute spawns (SpecChallengeActive).
