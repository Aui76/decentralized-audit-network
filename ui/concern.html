What verifying means

An audit is a public on-chain claim: a named tool ran on a pinned contract and posted a pass/fail result root.

Verify means you run the same tool on that same contract yourself — no wallet, no account. If your root matches what the chain recorded, the result reproduces. If not, something is worth a closer look.

This page never decides anything — the network only settles on a result you can re-run and reproduce yourself. So verify in section 1 above first, then answer the questions below.

Advanced The network verify tool itself may be wrong

This path is for when the problem is the network's shared verify tool — the registered harness everyone re-runs — not one mistaken result on a single audit row. You believe the tool itself is incomplete (it misses whole classes of bugs) or incorrect (it produces wrong roots against live bytecode), so any audit that depended on it may be untrustworthy.

If that is not what you mean, stay in the sections above:

  • You re-ran verify on this audit and got a different result. That is a claim or integrity path — use section 3.
  • Your worry sits outside what this audit declared it would check. That is spec-gap on this row's scope — still section 3, not a network-tool upgrade.

Structural upgrade is how the network repairs its own verify toolkit. You prove the canonical harness fails on O — the contract address locked in when the audit was posted — register a corrected tool F, and put that fix to jury adoption. Settled rows stay settled; what changes is which tool the network runs from here on.

Re-run verify in section 1 before you continue, unless you already have. Take this path only when the shared harness is the problem — not when you could challenge a single result through a normal claim in section 3.
