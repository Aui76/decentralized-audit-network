Each audit row is one pinned case: bounty in escrow, bytecode hash frozen at submit, spec declared at Gate A. The cell stores state and commitments only — tools run off-chain. This doc is the state machine: legal positions and transitions for one auditId.

Narrative framing: mechanism. Gate A detail: The door (Gate A). Break path: Vulnerability disclosure.

Three ways a row closes

Every audit ends in exactly one terminal posture:

End statePathMeaning
InBlockPositiveChallenge window elapsed clean; confirmAudit minted a positive block.
ExploitedBreakA proof-bearing claim reproduced FAIL on pinned bytecode; discoverer paid from escrow.
InvalidatedOverlayPre-mint void — Spec arbiter or Integrity review; no positive block minted.

Until one of these, the row is open.

Happy path vs break path

HAPPY (positive block)
  Submitted → Assigned → InAudit → AwaitingWindow → Audited → InBlock
              │            │              │
              │            │ proveFail    │ claimVulnerability (Gate C)
              │            ▼              ▼
              │         Claimed ──────────┘
              │            │
              │            ▼
              │        Exploited (if FAIL reproduces)
              │
OVERLAY (pre-InBlock void)
  Submitted … AwaitingWindow ──► Invalidated
  • `provePass` during InAudit opens the challenge window — the success path can still fail if Gate C fires before confirm.
  • `proveFail` during InAudit skips the window and enters `Claimed` immediately (assigned auditor as claimant).
  • Overlays can void the line any time before InBlock.
