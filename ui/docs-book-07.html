The promise needs a shape

Chapter 6 named three checks. Each ends the same way: a registered tool states pass or fail, and strangers disagree only by re-running what was run.

That only works if “what was run” is not a story. It has to be a fixed object two parties compute independently and get the same answer from.

That object is the fingerprint — a 32-byte digest. At the audit (Gate B) and the challenge (Gate C), the chain stores `resultRoot` under `AUDIT_RESULT_V1`. At the door (Gate A), `specPassDigest` under `AUDIT_SPEC_RUN_V1`.

What gets pinned

Every audit row pins four things:

PinMeaning
ToolRegistered check on-chain — not “best practice”
ArtifactBytecode pinned at submit — no moving target
SpecDeclared check from the door (Gate A)
Run contextCompiler, tool version — inside posted digest

Each gate compares the digest its phase owns. Run the tool with the inputs for that phase. If you cannot reproduce the digest for that gate, you cannot settle the dispute on that path.

What the chain holds

The cell does not execute tools on-chain. It stores commitments and moves state when independent re-runs match.

Value settles on the fingerprint. The cycle exists to produce it.

Before you turn the page

Fingerprint — 32-byte deterministic digest per gate phase.

Reproducible — same tool, same inputs → same digest.

Binary — pass or fail on-chain for each gate’s phase.
