Spec-gap overlay

Real fail proof landed on O, but S never promised to check P — how does the network record the fact without silently rewriting the within-spec PASS?

Story context

  • Book ch 12 · Overlays

Spec-gap records a reproducible FAIL on pinned contract O for property P when P was not part of declared spec S on that audit row. The overlay is non-destructive: the original “O satisfies S” result stays intact unless a separate within-spec path (Gate C) succeeds.

Within-spec misses → Vulnerability disclosure (Gate C). Wrong declaration → Spec arbiter (Gate A). Network-wide harness gaps → Structural upgrades.

Module: SpecGapModule.sol + SpecGapLib.sol (L1 satellite).

Spec-gap vs Gate C (routing)

The network does not infer scope from rhetoric. It reads which evaluator tool you cite:

Evaluator citedPathEffect on original row
Audit's declared `specToolId` (within S)Gate C — claimVulnerabilityDestructive — may → Exploited, failed++, dispute lane
Different canonical `isInvariantEvaluator` (outside S)Spec-gap — openSpecGapNon-destructive — row stays AwaitingWindow, Audited, or InBlock

Spec-gap filing does not pause the audit for within-S settlement, mark `Exploited`, or increment original auditor `failed++`.

Where it sits

Provisional PASS or InBlock on audit row (O, S on file)
        │
        │  FAIL witness on O — but evaluator ≠ audit's specToolId
        ▼
   openSpecGap  →  protocol concede | contest | decline | adopt
        │
        └── original within-S pass unchanged unless Gate C separately wins

May file while claim-eligible: AwaitingWindow, Audited, InBlock.

Overlay states

StatusMeaning
FiledFAIL witness + stake posted; protocol decision window open
ConfirmedProtocol concedes — witness violates P on O; stake returned; no discoverer payout yet
DeclinedProtocol declines relevance; record may still stand if witness was valid
FalseContest replay PASS on P; filer stake slashed
AdoptedProtocol paid discoverer via adoptSpecGap — only payout path
ExpiredAbandoned past claimResolutionWindow; liveness clear

Flow (typical)

openSpecGap(O, witness, evaluator, stake)
        ↓
  protocolConcedeSpecGap          → Confirmed (stake back)
  protocolContestSpecGap          → dispute replay → False or Confirmed
  protocolDeclineSpecGapRelevance → Declined
  confirmSpecGapSilence           → after protocol window; stake returned
  adoptSpecGap                    → Adopted (protocol-funded payment)
  expireSpecGap / expireSpecGapDispute → liveness

Contest settlement: evaluator replay via dispute path — confirmAudit → resolveFromDispute (same binary discipline as other tool paths).
