Mechanism

How** do the design constraints land in architecture — layers, loop, gates, and what may not move money?

Story: Book ch 2 — The idea

The Audit Cell replaces an audit opinion with a result anyone can reproduce: a named tool, a written spec, pinned bytecode, and a binary pass/fail committed on-chain. No firm desk, no committee, no discretionary payout.

Procedure depth: Flows book · layer doctrine: Modular architecture.

Thesis (one line)

Same tool + same pinned code + same spec on file → same answer for any party; pay and slash follow replay, not reputation.

Three layers (L0 / L1 / L2)

The stack splits so settled truth stays small and everything else can evolve without rewriting what the network already paid on.

LayerWhat it isWhat it may do
L0 — CellImmutable settlement coreAudit state, escrow, pass/fail commitments, challenge window → confirm, pay/slash by fixed rules
L1 — SatellitesWired modules beside the cellTouch settled truth through defined seams — spec void, integrity void, assignment draw, claim/dispute settlement, structural adopt
L2 — OrgansReplaceable readers and surfacesBob cards, phase tags, contagion, FMEA, membrane UI — alarm and present only

Full state machine: Audit lifecycle.

Two outcomes on one row

PathEnd state
PASSCode clears spec on file; survives window; auditor minted; tools credited toward canon
FAIL / exploitClaim + independent re-audit; if FAIL reproduces on pin → Exploited, discoverer paid from reserves (no fresh mint); non-reproducing claim rejected

Break path spec: Vulnerability disclosure.

Three gates (no fourth court)

Every audit decision routes through one gate question:

GateQuestionMechanism
A — SpecIs the declared check auditable and re-runnable?Declare at submit; re-run at accept; Spec arbiter voids bad specs pre-mint
B — VerdictDoes pinned code pass the spec on file?Auditor provePass / proveFail during InAudit
C — ClaimDid PASS miss a reproducible fail?Proof-bearing claim + dispute re-audit

Protocol reject after assign caps who audits — not whether the spec or verdict is wrong (Gate A vs Gate C).

Signal vs punishment

Off-chain hints (Bob card, contagion, phase tags) may alarm and route scrutiny. They do not auto-slash or auto-pay.

Punishment follows only when a registered tool produces a deterministic on-chain-verifiable result — integrity review, spec arbiter, or claim path (Audit mutex).

Primary product

The product is the toolkit: machine-checkable checks that canonize after verified on-chain use. Individual audits are how the set grows.
