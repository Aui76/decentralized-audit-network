Three questions, not three committees

Discipline question: Can a registered tool state the problem in binary form — pass or fail?

If yes → one of three checks. If no → out of scope for settlement.

CheckQuestion in plain English
The door (Gate A)Is the declared check valid before anyone audits?
The audit (Gate B)Does the code pass or fail that check?
The challenge (Gate C)After a pass, can someone prove the pass was wrong?

Three checks. No fourth appeals court on code truth.

Not here: overlay modules (spec arbiter, integrity review) — Chapter 12. Vocabulary for claimed — Chapter 5.

The door (Gate A)

Question: Is what the protocol declared — spec, binding, audibility — valid and machine-checkable?

Entrance check. Sloppy or non-auditable declaration stops here — void before rewards mint. “We meant a different spec” belongs at the door — not in a vulnerability claim.

Protocols declare at submit. Auditors re-run at accept. Still binary.

Wrong spec filed → this audit should never have counted.

The audit (Gate B)

Question: Does pinned code pass or fail the spec on file?

VerdictMeaning
Fail (`proveFail`)No challenge window; patch and submit fresh audit
PassChallenge window opens before success path (positive path) can close

The challenge (Gate C)

Question: After a pass, can a registered tool show deployed code fails the spec on file?

Proof-bearing claim: reproducible FAIL `resultRoot`, stake, eligible party.

OutcomeMeaning
Tool cannot reproduce failSettlement silent — no severity thread
Tool reproduces failClaimed → challenge procedure (break path) to completion

Not we meant a different spec (Gate A). Not website looks phishy (out of scope if spec never bound it).
