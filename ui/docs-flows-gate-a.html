The door (Gate A)

Story context

  • Book ch 2 · The idea
  • Book ch 3 · Toolkit as product
  • Book ch 6 · Three checks

Gate A is the entrance check on the declared spec — the machine-checkable filing that says what “pass” means for this audit. The cell never runs spec tools on-chain; it stores commitments and compares digests parties reproduce off-chain.

Question Gate A answers: Was the filing valid and auditable before anyone judged the code — and did the assigned auditor re-run the same filing at accept?

Wrong-spec fights (“we meant a different check”) belong here. Missed-bug fights after a pass belong on Vulnerability disclosure (Gate C).

Not here: pre-mint overlay procedure (challengeSpecInvalid, arbiter assignment, void hooks) — Spec arbiter.

At submit, the protocol posts bounty, pins bytecode, and declares spec S (via specHash + spec-validation tool + PASS digest). At accept, the drawn auditor must re-run the same tool on the same spec and match what was stored. Only then does Gate B (verdict on code) proceed.

States and transitions: Audit lifecycle.

Principle

Anyone with the exact spec artifact, exact registered tool version, and exact inputs must get the same PASS/FAIL and the same errorsRoot. The cell records specRunDigest(...) only — it does not execute tools inside AuditCell.

Declare at submit (submitAudit)

The protocol commits to off-chain spec S and to a PASS run of a registered spec-validation tool against S.

ParameterMeaning
specHashContent address of spec artifact S (what “on file” means for this audit)
specToolIdRegistered tool with isSpecValidationTool == true (how S is checked)
specErrorsRootOutput root from a PASS run of that tool on S (declared at submit)

The cell stores:

specPassDigest = specRunDigest(specHash, specToolId, pass=true, specErrorsRoot)

Reverts if specToolId is zero, unregistered, or not marked as a spec-validation tool.

Re-run at accept (acceptAudit)

The assigned auditor calls acceptAudit(id, specErrorsRoot) after re-running the same spec tool on the same specHash. The supplied root must match the stored specPassDigest.

OutcomeMeaning
Digest matchesAuditor attests the declaration is reproducible; audit moves toward Gate B
MismatchSpecRunMismatch — declared filing cannot be reproduced as stated

Digest function (deterministic binding AUDIT_SPEC_RUN_V1):

specRunDigest(specHash, specToolId, pass, errorsRoot)

Pre-mint spec challenge (Gate A overlay)

After intake, while the line is still pre-mint (Submitted … AwaitingWindow), a permissionless challenger may dispute the declared spec PASS binding — not “the code failed the spec on file.”

If sustainedEffect
Declaration invalidLine → Invalidated; bounty returned (per module rules); artifact pin cleared

That fight is Gate A, not Gate C: should this audit line have been admitted? Procedure: Spec arbiter.
