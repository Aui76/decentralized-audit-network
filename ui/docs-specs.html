Specs & checking

How a contract is checked — declared up front, re-run to accept, reproducible by anyone.

What a spec is

A spec is a written, machine-checkable statement of what a contract must do, paired with a named tool that returns one answer: pass or fail. The spec itself lives off-chain; only a fingerprint of it is recorded — so anyone holding the spec can prove it's the exact one the audit used.

Declared up front

When a contract is posted, the protocol names the spec and the tool, and what a passing run of that tool produces. This is fixed at that moment and can't be changed afterward. It is the ground truth for the audit.

Re-run to accept

Before an auditor takes the job, they run the same spec tool themselves and must get the same result the protocol declared. If it doesn't match, they can't accept. The check is agreed by reproduction — not by trust.

Reproducible by anyone

The network never runs the tool itself. It records the commitment, holds the bounty, and times the window. The checking happens off-chain, where anyone running the same tool on the same code gets the same answer. That is what turns a verdict into a proof.

The tools

Tools are registered openly: protocols name them, auditors run them. A tool earns canonical standing by surviving real audits — not by a vote. That growing set of trusted tools is the network's real product.
