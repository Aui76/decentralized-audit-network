Ripe-slot indicator — deferred beacon

Where might the next verification primitive want to be built** — without minting or gating?

Status: DEFERRED — exploratory design note only. No indexer, no CI, no card field today.

Source: dan-core ripe-slot-indicator-design-note.txt

What it is

A probabilistic beacon that ranks vulnerability classes by where the next high-value verification primitive “wants to be built” — at the intersection of:

  • Demand — existing tools have saturated on class c (marginal gain → 0) but exploits still escape (residual demand, boosted by contagion R₀).
  • Supply — neighbouring classes in the co-occurrence graph already have canonical tools (“adjacent possible” exists).
D(c)         = saturation(c) × residualDemand(c)
S(c)         = fraction of c's neighbours with a canonical tool
RIPENESS(c)  = D(c) × S(c)

High ripeness + high fan-out → predicted age (broad primitive). High ripeness + low fan-out → predicted improvement (narrow stirrup).

Computation is event-driven (dyad splits, tool-P2G triangles, contagion spikes) — a maintained view over the alarm stream, not a periodic scan. Same thermodynamic spine as Network flow; applied to tools via FMEA gap vectors, not auditors.

Complement to issuance: the depression floor keeps activity (FLOW) alive; ripe-slot points builder attention at where stock risk concentrates. Orthogonal to Adaptive issuance — Uncapped Supply by choice.

Guardrail — beacon, not budget

Per the design note §0 and Settlement vs signal:

Integrator rules (today and if it ships)

  • Do not quote APY, mint size, or discoverer payout from ripeness rank.
  • Do not show ripe-slot on retail/investor Bob cards unless explicitly labeled R&D / maintainer — not settlement.
  • Do not treat a “top ripe” class as proof of safety or unsafe — it is a where to build / where to attack hint only.
  • Do keep wallet flows on chain state; use Bounty bands and Assurance tiers as the only economic card signals today.
