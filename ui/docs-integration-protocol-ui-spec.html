Core rule

The network certifies one pinned audit case per auditId — not a protocol brand or wallet.

WrongRight
“Acme Protocol ✅ audited”“0xBBB — Audit #47 — PASS — case root 0xabc… (pinned at submit)”

Each row is authoritative only for its (deployedAddress, artifactHash at submit, spec binding) triple.

Grouping under one protocol

Multiple rows coexist in history. Each row covers only its own deployment + pinned hash.

Protocol 0xBEEF
├── #91  0xCCC  PASS (InBlock)     ← product v2
├── #47  0xBBB  PASS (InBlock)     ← product v1
└── #12  0xAAA  Exploited          ← legacy

“Current product” (off-chain convention): e.g. latest InBlock PASS for the address the protocol advertises — label “Latest listed deployment”, not “Protocol audited.”

Version upgrades (new address / new bytecode)

New deploy → new audit, full cycle. Prior rows stay on chain.

v1v2
#12, 0xAAA, artifact H₁#47, 0xBBB, artifact H₂
Historical PASS rowNew Gate A + queue + PASS + window

UI copy: “Deployment 0xAAA — PASS (historical)” vs “Deployment 0xBBB — PASS”. No “upgraded certification” unless a lineage link is shown (below).

Proxy / implementation staleness

On submit: artifactHash = deployedAddress.codehash (then). After upgradeTo, live code may differ.

Required client check before a green badge:

liveHash   = extcodehash(deployedAddress)
pinnedHash = audits(auditId).artifactHash

if liveHash != pinnedHash → STALE_IMPLEMENTATION
else if state == InBlock  → CERTIFIED_FOR_PINNED_BYTECODE (qualified)
StatusBadgeCopy
STALE_IMPLEMENTATIONAmber“Implementation changed since audit. Re-audit required.”
Match + InBlockGreen (qualified)“PASS for bytecode pinned at submit”

Do not show a green check on address alone for proxy contracts without this check.

Upgrade lineage — supersedesAuditId

Purpose: Product navigation. Does not inherit PASS, void prior audits, or change prior settlement.

Set at submit: submitAudit(..., supersedesAuditId = priorId) on ordinary audits only.

On-chain rules (`CellLogicLib.createAuditExt`):

RuleBehaviour
Same protocolSubmitter must equal audits[priorId].protocol
Prior existspriorId < newId; prior must not be a fix audit
New artifactNew caseRoot / artifact binding must differ from prior
Fix auditssupersedesAuditId must be 0 on submitFixAudit
DedupSame bytecode still reverts CaseAlreadyAudited / ArtifactAlreadyAudited — supersedes is not a dedup bypass

Does not happen on lineage link:

  • Prior PASS does not cover the new deployment
  • Prior row is not voided or upgraded
  • New row does not skip Gate A, assignment, or window

Integrator “is this call safe?” checklist

  1. Which deployedAddress is the user touching?
  2. Find audit(s) where auditDeployedOf(id) matches and auditStateOf(id) == InBlock (or policy: allow window states with warning).
  3. Staleness: extcodehash(deployed) == artifactHash. If not → not safe.
  4. Multiple PASS rows → match this address + hash, not “latest protocol audit.”
  5. Claimed or Exploited on that id → not safe.
  6. Disclose active Claimed even if original PASS mint stands.
