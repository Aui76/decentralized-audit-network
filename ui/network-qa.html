What is Audit Cell?

Ten plain questions that, together, describe what this network is — why it exists, how trust works, what one audit means, and where the honest limits are.

Opinions are cheap. Reproduction is the currency.

This page is about the network. Worried about a specific audit? Use the explorer → Find your step on concern.html — that router is audit-scoped, not a general FAQ.

1What problem is this network solving?

Everyone · start here

Smart contracts cannot be patched after deployment. If the code is wrong, the loss is permanent — so getting it right before launch is not a nice-to-have; it is survival.

The usual answer today is the professional audit: a protocol pays a firm, waits weeks, and receives a report. That model has three built-in weaknesses the network is designed to replace:

  • The auditor is paid by the audited. Incentives bend toward verdicts that do not create friction for the client.
  • No stake after the report. If a missed bug is exploited later, the audit fee is not returned and the finder is not automatically funded by the mechanism.
  • The result is not reproducible. “Audited by Firm X” is an opinion you must trust — not a check you can re-run on live bytecode and compare.

Audit Cell (the Decentralized Audit Network on Base Sepolia testnet today) replaces that opinion with a binary pass/fail result from a named tool against a written spec, posted on-chain as a fingerprint anyone can recompute. The goal is public audit infrastructure — not another PDF brand.

2What makes a result trustworthy if I don't know the auditor?

Verifiers · explorers · skeptics

Trust does not come from reputation alone. Each settled audit posts a result root on-chain: a compact fingerprint that means “this registered tool ran on O — the original target contract pinned when the audit was posted — and produced this pass/fail outcome.”

Verify means you run the same tool on that same contract address yourself — no wallet, no account, no permission from anyone. If your recomputed root matches what the chain recorded, the result reproduces. If it does not match, something deserves a closer look before you treat the pass as settled fact.

That is the core trust model: reproduction, not authority. The network does not ask you to believe a firm, a committee, or a dashboard. It asks you to re-run the check. Strangers can do this read-only; only escalation paths (claims, posting bounties, pool registration) require a wallet and stake.

Verify proves the on-chain binding and encoding story matches live bytecode. It does not by itself re-execute the full EVM from scratch in your browser — for that depth you use the local harness documented in the verify guide. Still, for most readers, “does my root match?” is the honest first gate.

3What does one audit actually promise — and what does it not promise?

Protocols · readers · anyone interpreting a pass

One audit is deliberately narrow. It promises exactly this: a named tool was run against a declared spec on the bytecode at O — the target contract address pinned at submit time — and the posted verdict (pass or fail) is the outcome of that run, challengeable under the network rules.

It does not promise:

  • That the whole protocol is “safe” or that users cannot lose funds in ways outside the spec.
  • That future upgrades, proxies, or off-chain components are covered unless they are part of the declared check.
  • That frontend/UI issues, governance politics, or legal compliance were reviewed.
  • That the declared spec was the right spec for what you care about — only that the contract was checked against the spec that was declared.

A flawless tool run against a hollow or mis-scoped spec returns a hollow verdict. The machine settles whether the contract failed that check — not whether the check was adequate for your risk model. Choosing scope is a protocol responsibility at post time; challenging scope belongs at Gate A, not inside an ordinary “they missed my bug” claim.

4Why only pass or fail — no "medium" or "informational"?

Auditors · dispute thinkers

The settlement core is built for machine-checkable questions. Either the contract fails the declared check, or it does not. There is no on-chain severity ladder, no partial pay for “kind of wrong,” and no appeals court that rewrites a verdict because humans disagree about priority.

That is a feature, not an omission. Severity debates, triage, and “is this in scope for my product?” belong in markets, reputation, and Gate A spec fights — not inside a single binary railroad. The network settles one thing at Gate B: pass/fail against the declared spec on the bytecode at O — the pinned original target.

When something more nuanced is needed — “we meant a different spec,” “this tool cannot express the property,” “the pass missed a real bug on the bytecode at O” — the three-gate model routes those to different mechanisms instead of smuggling them into severity labels on one verdict.

5What are the three gates?

Everyone who might escalate · essential map

Most confusion comes from bringing the wrong kind of fight to the wrong door. The network separates disputes into three gates.

What is O? O is the original target contract: the deployed address the protocol pinned when it posted the bounty. Every verify run, claim, and dispute re-runs the registered tool on the bytecode at that address — not on a patched deployment, a replacement contract, or code upgraded elsewhere unless the audit explicitly bound to it.

GateQuestionTypical path
AIs the declared spec auditable / the right spec?Spec declared at submit; challenge + arbiter before the audit mints as canonical work
BDoes the code at O pass or fail that spec?Auditor runs registered tool on O → posts pass/fail root
CWas a pass wrong on the code at O?Staked claim during challenge window → independent re-check on the same pinned address O

Rule of thumb: “They checked the wrong thing” → Gate A. “They said pass but I can reproduce a fail on O with the same tool” → Gate C. Vague worries about frontend copy or off-list risks may be out of scope for an ordinary claim — read the declared scope list on the audit record first.

Protocol choice at post time decides who audits and what spec/tool pair is bound — not whether strangers can later reproduce the posted root.

6Why is auditor pay delayed until after a challenge window?

Auditors · protocols · anyone asking “why wait?”

An auditor is not paid the moment they submit a verdict. Payment comes at confirmation, after the challenge period closes with no successful bug claim on that pass. That delay is adversarial by design — not bureaucracy.

During the window, anyone who can reproduce a mismatch can file a staked claim. The original pass is provisional until strangers had time to re-run verify on O and, if they find a real fail root on that pinned bytecode, force resolution. Pay ties to accuracy under challenge, not to delivering a report on deadline.

For protocols, the window is the interval where “passed” still means “surviving scrutiny.” For auditors, it is the stake: your reward depends on the contract at O surviving that scrutiny, not merely on filing a green checkmark. Missed bugs that reproduce on O can void the happy path and fund the finder from reserves.

7How does someone challenge a pass they think is wrong?

Bug-hunters · pool auditors · skeptical verifiers

During the open challenge period on a pass, a qualified participant can file a claim — not a vague accusation, but a proof path: you believe the auditor missed a bug that reproduces under the pinned tool on O, the same target address bound when the audit was posted.

Filing requires stake in AUDIT. If the claim succeeds, the original audit is marked failed/exploited and the finder is paid from reserves — never from fresh mint on the failure path. If the bug cannot be reproduced on O, the claim is rejected and the stake is at risk.

Settlement is always on O — the original target contract pinned at post, not a patched copy or a new deployment. Patching the code, deploying a fix elsewhere, or finishing some off-chain procedure does not settle a claim for you. Same address, same bytecode binding, same tool, same reproducible result — or the claim fails.

If a claim is filed but never taken up and its window expires, coordination risk can slash the finder’s stake with no payout — filing is serious. Read the bug-hunter guide before staking.

8What is the AUDIT token for — and how do people earn it?

Auditors · protocols · token readers

AUDIT is the network’s unit of account for bounties, stakes, and pool participation — not a yield product or equity claim. Protocols lock escrow in AUDIT to post bounties. Claimants stake AUDIT to file serious challenges. Auditors hold position-scaled amounts to register in the assignment pool (early slots can start at zero; later slots expect tokens earned by doing real work).

How tokens enter circulation: primarily through confirmed audits — activity-led mint when work survives the challenge window. That ties supply growth to verified mechanism use, not to marketing airdrops or inflationary exploit rewards.

How bug finders get paid: from reserves — a slice of each confirmed audit reward — when a claim reproduces a fail on O, the pinned original target contract, not on code deployed or upgraded elsewhere. Failures draw down reserves; they are not rewarded with fresh supply as a consolation prize.

The token pays for the railroad — escrows, stakes, pool holds — and records that real audits happened. Legal classification in your jurisdiction is your problem; the docs describe function, not investment promises.

9What's the real product — one audit, or something bigger?

Long-term thinkers · tool authors · protocols

Any single audit matters — but the long-term product is the shared toolkit: a growing library of machine-checkable specs and registered tools that strangers can re-run without asking permission. Individual audits are how the library grows; canonical status comes from real use under stress, not from a committee blessing a PDF.

Incumbent audit firms sell trust and narrative. This network sells public history plus reusable checks. A pass on audit #N is a row in that history; the tool and spec behind it are bricks in infrastructure the next protocol can reuse or challenge at Gate A.

That is why “ideas become real” only when a stranger can re-run them and get the same answer with the result on-chain. Talk — concern clicks, want-board signals, forum posts — proves nothing alone. Reproduction recorded on-chain is the currency.

10What is this network not — and where is it today?

Honesty block · read before trusting with real money

Testnet · Base Sepolia — test funds only

Today this is reconstructed cell infrastructure on testnet with a live membrane UI (explorer, verify, manage, concern router). It is not a finished mainnet product, a legal/compliance service, or a guarantee that passed contracts cannot be hacked in ways outside the declared spec.

What we will not claim:

  • Zero risk or guaranteed token returns.
  • Replacement for your legal, regulatory, or enterprise security duties.
  • That interpretation UI (concern router, want-board rankings) settles anything — those tools route and alarm only; payout and slash always go through on-chain verify + staked paths.
  • That admin parameters are immutable forever — economics can be tuned today; multisig/timelock hardening is part of the roadmap before large public bounties.

If something looks wrong on a specific audit, use the explorer → Find your step walkthrough — it reads chain facts for that audit and suggests one next step. It does not send your clicks to the chain and cannot slash you for a wrong suggestion.
