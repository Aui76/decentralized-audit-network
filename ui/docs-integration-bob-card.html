Bob card

What JSON does deriveBobCard emit** — and which inputs are production MUSTs?

Story context

  • Book ch 22 · Reading the chain

Presentation layer: turn on-chain facts + integrator inputs into a card humans can read. The cell emits state and events; `cell/indexer/deriveBobCard.mjs` composes hints and outcomes off-chain.

Pipeline

Cell / network facts + events
        ↓
Hint tools (PASS | FAIL each)     ← manipulation / assurance axes (HINT_IDS)
        ↓
Composer (e.g. **defended-spec-on-wash-line** / `P5d-P2G`)           ← optional multi-hint signals
        ↓
Bob card (outcome · shape · action · maturity · phases · flow)
        ↓
Optional attachments              ← scope, spec-quality, contagion (helper modules)

Each hint tool answers one narrow question. PASS or FAIL only — not “audit is safe.” Composers combine hints; `deriveBobCard(input)` returns the card JSON integrators render.

Decision trees

Normative mapping implemented in deriveBobCard.mjs. Golden rows lock these paths — see #fixtures.

Outcome (lifecycle base)

specChallengeActiveOPENALARM

Hint downgrade (CLEAN only)

Any FAIL on: poison-tool, lazy-spec, trust-dissolve, P2G, code-unverified, spec-unverified, P5d → downgrade CLEAN → WEAK.

Action

  1. `P5d-P2G` FAIL → WALK (with integrityReviewSuggested when bounty in top quartile).
  2. `P2G` FAIL → P2G.suggestedAction (often WALK on exploit tail).
  3. Else `deriveAction(outcome, confidence)`:
OutcomeConfidenceAction
VOID, BROKENanyWALK
OPEN, WEAKanyWAIT
CLEANHIGHINVEST
CLEANMED/LOWWAIT

Contagion fuse (portfolio Phase B)

When outbreak exposure hits the audit:

  • CLEAN → WEAK
  • INVEST → WAIT
  • Confidence capped to LOW (fuse) or MED (advisory)
  • Pattern tag portfolio-fuse appended

Confidence (sketch)

  • Base from bobChecks: all VERIFIED → HIGH on CLEAN; ONCHAIN-only slots → MED; WEAK → LOW.
  • Trust scores (specToolTrust, auditorDyadTrust, protocolLineTrust) adjust up/down.
  • Single-protocol CLEAN capped at MED (O1-self-tool path).
  • Alarm shape or P5d/P2G/trust-dissolve hints → LOW cap.

Action rules: #decision-trees.

O1 — ordinary CLEAN

Input sketch: state: InBlock, specAuditorAttested: true, all bobChecks VERIFIED, multi-protocol tool trust.

Expected: outcome: CLEAN, action: INVEST, patterns: ["L0"], maturityBand: warm.

O1-prime variant: omit bobChecks → WEAK / WAIT / H-pass — indexers must not emit CLEAN+HIGH on ONCHAIN-only code slot.

O1-scope — scope manifest

Spec: fixtures/spec-reference-v0.json via specReferenceFixture.mjs.

Expected scope: scopeStatus: has-categories, stable scopeRoot, covered/not-checked counts locked in deriveScope.test.mjs. No coverage score.

O1-defended-p2g — defended spec on wash line

Input: defended spec challenge + wash-triangle stats + bounty in top quartile.

Expected: action: WALK, integrityReviewSuggested: true, composed hint `P5d-P2G` (defended-spec-on-wash-line). Low-bounty variant drops integrity suggest only.

S1 — structural OPEN + JURY

Input: gap in Probation, jury not adopted; patterns S0-pending, SJ-open.

Expected: outcome: OPEN, action: WAIT, filer modifier INDEP-GAP in default test filer path.

Adding fixtures

Add a row only when:

  1. Outcome, shape, or mandatory alarm mapping changes, and
  2. A reproducible snapshot exists (Foundry test in cathedral, or deterministic input object in cell).

Do not add fixtures for UI copy-only tweaks.
