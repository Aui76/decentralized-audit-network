Scope — claimed check categories

Off-chain scope manifest from the Gate A spec artifact. No on-chain `scopeRoot` in the cell — v1 is indexer + writer convention; pinning deferred until real spec-tool usage converges.

Facts vs interpretation

"Covered" and "not checked" on a Bob card are reproducible tags over declared invariants — not a depth score and not a network opinion. Same discipline as Bob card.

Writer shape (Gate A artifact)

Each invariant carries tagged scope symbols:

{
  "invariants": [
    {
      "id": "swap-rate-fixed",
      "statement": "swap output follows declared rate",
      "scope": ["SWAP-RATE", "BALANCE-ACCOUNTING"]
    }
  ]
}

Symbols must appear in the v0 taxonomy (#taxonomy · indexer/scopeTaxonomyV0.mjs).

Deterministic manifest

From tagged invariants the indexer derives:

  1. Sorted unique categoryId set (order-independent).
  2. scopeRoot = keccak256(abi.encodePacked(sorted categoryIds)).

Same invariant order in the JSON file must not change scopeRoot.

Taxonomy — v0 seed categories {#taxonomy}

Foundation vocabulary for scope[] tags on Gate A invariants. Seed set, not a whitelist — extend only with dry-run evidence (#taxonomy-dry-run); canonical status is earned by cross-protocol usage.

Registry source: `cell/indexer/scopeTaxonomyV0.mjs` (V0_SCOPE_SYMBOLS).

How to read an entry

  • User-question first — each symbol maps to a plain worry a user actually has.
  • Covered = invariant checks — tag only when an invariant checks that worry, not when marketing mentions it.
  • Disjoint by intent — "Not this" boundaries keep tags from overlapping.

Identifier grammar

DAN:SCOPE:v0:{SYMBOL}

Helper: categoryIdForSymbol(symbol) in deriveScope.mjs (uses danIdV0.mjs).

Taxonomy dry-run {#taxonomy-dry-run}

Pressure-test #taxonomy against a real contract archetype before changing code or on-chain vocabulary.

Rule: facts not judgment; covered = checked at least once, never depth.

Method (5 steps)

  1. Pick one archetype — concrete contract shape (not "DeFi in general").
  2. List 6–10 invariants — checkable statements an auditor would ask.
  3. Tag each to one category — use Not this boundaries.
  4. Flag failures: GAP · FORCED DOUBLE-CLAIM · NAME/SCOPE SKEW.
  5. Record validations — boundaries that worked.

Fix order (lightest first)

FindingAction
GAP (disjoint concern)Add category with Means / Not this
NAME/SCOPE SKEWBroaden + rename existing category
FORCED DOUBLE-CLAIMClarify Not this on both categories
Coarse category hides distinct FMEA gapsSplit via parentId (FMEA signal required)
Clean tagRecord validation only

After dry-run: edit taxonomy table above only unless renames force fixture sync. Run npm run test:organs-o4 in cell/indexer/.

Quality — linter vocabulary

Off-chain linter + Bob presentation layer. No contract, no aggregate score, no auto-invalidate on these labels alone — binary spec dispute stays Gate A + challengeSpecInvalid.

Pair with #scope (what categories are claimed) and Bob card § integrator-surface (specQualityLine on the card).

The gap this fills

LayerWhat the cell already does
Gate ARegistered spec tool PASS → specPassDigest on-chain
Binary disputechallengeSpecInvalid → arbiter → Invalidated
Scope v1Lists claimed categories — not vacuity, mis-tags, or marketing honesty

Spec-quality instruments name the middle ground: machine-PASS specs that are vacuous, mis-tagged, thin, or oversold.

v0 classes

SymbolPlain meaningVoids audit today?
SPEC-INVALIDTool FAIL or arbiter FAILYes (on-chain)
SPEC-INAUDITABLECannot re-run to same digestYes (blocked / challenge)
SPEC-VACUOUSPASS but invariant rules nothing meaningfulNo — warning
SPEC-MIS-SCOPEDscope[] does not match what statement checksNo — warning
SPEC-THINHonest but narrow — large not-checked listNo — disclosure
SPEC-MARKETING-SKEWOff-chain copy asserts checks JSON lacksNo — integrator alarm
SPEC-STALELive bytecode ≠ pinned artifactNo — new audit

SPEC-THIN is disclosure, not fraud. SPEC-VACUOUS, SPEC-MIS-SCOPED, SPEC-MARKETING-SKEW are integrity problems.

Bob card behavior

When specInvariants are present, deriveBobCard attaches spec-quality copy via deriveBobSpecQuality. Does not change outcome, action, or confidence — facts layer only.

Integrator warnings for missing scope (SCOPE_MANIFEST_UNAVAILABLE, SCOPE_UNKNOWN_SYMBOL) are separate from spec-quality findings.

Outcome correlation

deriveSpecQualityCorroboration compares linter findings (M) to deterministic audit outcomes (A) after the window closes — validates the instrument, does not re-judge the audit.

deriveSpecQualityFalseNegative fires when the linter was integrity-clean but a falsifying fact appears (spec invalidated, pain in covered scope, or manual maintainer filing). Maintainer/fleet only — not investor-facing card copy.

Fleet JSONL append/report (specQualityFleetLog.mjs) is O7 — see #fleet.

Fleet — outcome correlation log

Fleet-scale spec scoring — advisory, does not gate settlement.

Append-only JSONL fleet log for outcome correlation: one closed audit → one row. Validates validateSpecQuality heuristics (M) against deterministic audit outcomes (A). Maintainer / fleet view — not investor Bob card fields.

Pair with #quality · #scope · The door (Gate A) · Spec arbiter.

Modules (cell/indexer/)

ModuleMeaning
specQualityFleetRecord.mjsSnapshot → fleet row (also used by O4 false-negative test — one module)

Deps (O4/O1): validateSpecQuality, deriveSpecQualityCorroboration, deriveSpecQualityFalseNegative, deriveScopeFromSpec, fmeaScopeBridgeV0, danIdV0 (FMEA-SCOPE ids).

When to run

  • After ≥10 production or testnet audits with closed windows.
  • Before changing linter regex thresholds or adding a SPEC-* class.
  • Not on a single audit to auto-tune rules (Goodhart guard — hold linter version fixed across periods).
